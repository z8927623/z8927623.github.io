Objective-C Runtime之消息

发表于 2016-02-03   |  

Method&Message

什么是Method？

方法就是一个类中有名字的一串代码，平常我们写的
- (int)func:(NString *)str { ... }就是一个方法，但是它在编译的时候会被转换成C函数调用，
static int func(SomeClass *self, SEL _cmd, NSString *str) { ... }，也就是说，所谓的方法实际上就是C函数。

从static int func(SomeClass *self, SEL _cmd, NSString *str) { ... }就可以明白为什么我们可以使用self和_cmd，因为他们在源码中是隐藏的，self代表着receiver，_cmd代表调用选择器selector。

什么是Message？

消息是Objective-C中的核心，必须借助于runtime系统才能完成。当我们不了解runtime的时候，一个简单的[receiver message]往往会被认为是函数调用，实际上它仅仅是在编译期间确定了要向receiver这个接收者发送message消息，而receiver如何响应这条消息，编译阶段是无法知晓的，这个要到运行的时候来判断了。

平常我们调用的[self func:@"呵呵哒"];会在编译期间被转化为objc_msgSend(self, @selector(func:), @"呵呵哒";，也就是说，OC中发送消息会被转化为objc_msgSend。

那么，我们是否可以直接调用objc_msgSend来发送消息呢？理论上是可以的，虽然苹果极力不推荐。实际上编译器为了更好地进行错误判断，会执行严格的类型检查，也就是说，如果我们直接编写以上代码，会通不过编译，假如我们非要这么做，可以将编译选项Enable Strict Checking of objc_msgSend Calls设置为NO，或者将代码修改为((int (*)(id, SEL, NSString *))objc_msgSend)(self, @selector(func:), @"呵呵哒";，就可以了。

objc_msgSend

objc_msgSend可能是iOS app运行时被调用的最多的函数了，那么它都干了什么？很可惜，为了提高调用速度，它是用汇编实现的。不过我们仍然可以通过源码来窥知一二。我们打开runtime源码然后搜索.s，.s是汇编文件的后缀

通过汇编大致可以知道objc_msgSend的具体步骤：

  1. 检查selector是否可以忽略

    • 可以忽略，返回self
    • 不可以忽略，跳转到2
  2. 检查消息接收者是否是nil
    • 不是nil，跳转到3
    • 是nil，直接返回0
  3. 手先从缓存中查找selector
    • 如果缓存命中，则执行它的imp
    • 如果没有找到缓存，跳转到4
  4. 先从本类的分发表中找
    • 如果找到，则执行它的imp，同时以selector为key，imp的地址为value存入缓存
    • 如果没找到，则通过isa指针继续查找其父类的分发表直到根类NSObject
  5. 如果到NSObject还没有找到，则执行动态方法决议

  6. 动态方法决议是个很重要的流程，接下来会重点讲

动态方法决议（Dynamic Method Resolution）

上面提到了，当到NSObject还没有找到selector的时候，就进入了动态方法决议，简单的说就是当我们无法找到匹配的方法实现时，runtime允许我们在运行期间动态地添加一个方法。

我们主要做了以下几件事

  • 实现+resolveInstanceMethod:或者+resolveClassMethod:
  • 检查selector是不是我们想要处理的方法
    • 匹配，则通过class_addMethod添加一个方法并返回YES，可以看到添加的方法是一个简单的C函数
    • 不匹配，调用super处理
  • 注意如果我们不提供一个实现直接返回会抛出NSInvalidArgumentException异常

消息转发（Message Forwarding）

如果动态方法决议还没有解决，则进入了消息转发流程，可分为快速转发和标准转发。

快速转发（Fast Forwarding）

通过快速转发我们可以将消息快速地转发给另一个对象，或者忽略，但是我们无法在快速转发中改写消息。

  • 实现-forwardingTargetForSelector:
  • 检查selector是否是我们想要处理的方法
    • 匹配，转发给能够处理该消息的对象
    • 不匹配，调用super处理

标准转发（Normal Forwarding）

快速转发之后就是标准转发，通过标准转发我们可以完整的实现转发流程，在错误之前给了我们最后一次机会。

runtime系统首先会调用-methodSignatureForSelector:去获取方法签名，然后使用方法签名去创建NSInvocation对象，它封装了原始消息和参数；接着-forwardInvocation:被调用，使用新创建的NSInvocation作为参数，默认的-forwardInvocation:会发送-doesNotRecognizeSelector消息抛出异常，因此我们必须覆写-forwardInvocation:，同时也要覆写-methodSignatureForSelector:去生成一个方法签名。

错误

如果动态方法决议和消息转发都没有解决，那只好抛出异常。runtime会发送-doesNotRecognizeSelector:消息，接着抛出NSInvalidArgumentException异常，整个过程也就结束了。

Objective-C Runtime之概览

发表于 2016-02-01   |  

什么，你和我说runtime，大多数iOS程序员听到runtime的时候，脑子里第一反映便是，这货太特么抽象难理解了，而且工作中不常用到。的确，我也是这么认为的，Objective-C里两大最难的点当属Runtime和Runloop。然而，难虽难，这硬骨头还是要啃啊，不然理解不到这门语言的精髓呀。

我们都知道，Objective-C这门语言是动态的，是扩充于C的面向对象语言，所谓动态的核心就是Runtime。平常写程序的时候，它被系统加载进去，然后在幕后默默工作着，我们甚至察觉不到它的存在。但是，假如没有它，OC就运行不了了。也就是说，C + Runtime = Objctive-C。

runtime.h

runtime是开源的，我们可以在https://opensource.apple.com/tarballs/objc4/下载。

runtime.h里声明了以下开头的函数，我们通过观察类的命名可以发现：

  • objc_...
    和runtime一些属性设置有关。比如
    • 类相关
      创建一个类（objc_allocateClassPair）、注册该类（objc_registerClassPair）、销毁该类（objc_disposeClassPair）、指定类名返回该类（objc_getClass）、指定类名返回该类元类（objc_getMetaClass）、获取注册的类的列表（objc_getClassList或者objc_copyClassList）等
    • 对象相关
      设置关联对象（objc_setAssociatedObject），获取关联对象（objc_getAssociatedObject）等。
    • 协议相关
      创建一个协议（objc_allocateProtocol）、注册该协议（objc_registerProtocol）、指定协议名字获取协议（objc_getProtocol）、运行时所有的协议名（objc_copyProtocolList）等。

  • class_...
    和类相关。
    比如获取类名（class_getName）、为类添加一个新的方法（class_addMethod）、获取类对应选择器的方法描述Method（class_getClassMethod）、获取类的变量列表（class_copyIvarList）、属性列表（class_copyPropertyList）、协议列表（class_copyProtocolList）、方法列表（class_copyMethodList）等。

  • object_..
    和对象相关。
    比如获取对象的类名（object_getClassName）、获取对象某个变量的值（object_getIvar）等。

  • method_...
    和方法描述有关。
    比如获取方法名SEL（method_getName）、方法的实现IMP（method_getImplementation）、参数和返回参数的类型编码（method_getTypeEncoding）等。

  • ivar_...
    和变量有关。
    比如获取变量名（ivar_getName）、实例变量的类型编码（ivar_getTypeEncoding）等。

  • property_...
    和属性方法有关。
    比如获取属性名（property_getName）、属性的编码（property_getAttributes）等。

  • protocol_...
    和协议有关。
    比如获取协议名（protocol_getName）、协议的方法描述（protocol_getMethodDescription）、判断某个协议是否实现另一个协议（protocol_conformsToProtocol）等。

  • sel_...
    和方法选择器selectors有关。
    比如获取选择器的名字（sel_getName）、注册一个选择器（sel_registerName或者sel_getUid）等。

  • imp_...
    和方法的实现有关。
    比如用block实现某selector的具体实现（imp_implementationWithBlock）等

foundation.h

Foundation是一个framework，它定义了OC类的基础。其中，NSObject类是Cocoa里大部分类的基类，它定义了一些基础的方法，比如description方法，NSObject对这个方法的实现仅仅是返回对象的名字和地址，我们可以通过覆写该方法来返回更多的信息。

除了实现了一些基本的方法，NSObject也定义了部分的runtime接口：

  • - (Class)class;
  • - (BOOL)isKindOfClass:(Class)class;
  • - (BOOL)isMemberOfClass:(Class)aClass;
  • - (BOOL)respondsToSelector:(SEL)selector;
  • + (BOOL)instancesRespondToSelector:(SEL)aSelector;
  • - (id)performSelector:(SEL)selector;
  • - (IMP)methodForSelector:(SEL)aSelector;
  • + (IMP)instanceMethodForSelector:(SEL)aSelector;
  • - (void)doesNotRecognizeSelector:(SEL)aSelector;
  • - (BOOL)conformsToProtocol:(Protocol *)aProtocol;
  • + (BOOL)conformsToProtocol:(Protocol *)protocol;
