额...runloop

发表于 2016-02-05   |  

run loop的目的是让我们的线程保持繁忙，同时当线程没事儿可做的时候使之休眠。

主线程的run loop已经在程序启动的时候作为启动进程运行，而子线程，我们需要手动使run loop运行。

run loop就像它的名字一样，是个循环，那它是什么循环呢，和线程有关，线程利用它去响应事件的

run loop接收两种类型的事件源：

  • 输入源（Input sources）：投递异步事件，通常是从另一个线程或者另一个app过来的。
  • Timer源（Timer sources）：投递同步事件，通常出现在timer运行的时候

下面这张图是run loop的运行状态图

Input sources投递异步事件去响应handlers，并且使runUntilDate:方法退出
Timer sources投递timer事件处理但是不会造成run loop退出

除了处理输入源，run loop也可以对自己的行为生成通知，我们可以注册run-loop observers，然后我们就可以收到这些通知，然后在线程上做一些自己的事情，我们可以使用Core Foundation去安装观察者在我们的线程上。

Run Loop Modes

当我们显式或者隐式运行run loop，都必须指定一个model，当指定mode之后，只有和该mode相关的事件源才能被投递到线程去处理，其他不相关的事件源只能被挂起，直到run loop回到和该事件源相关的mode。

Mode Name Description
Default NSDefaultRunLoopMode(Cocoa) kCFRunLoopDefaultMode(Core Foundation) 默认模式，大部分情况我们使用这个模式就OK了
Connection NSConnectionReplyMode(Cocoa) 这个模式是给NSConnection处理网络回调用的，我们几乎用不到
Modal NSModalPanelRunLoopMode(Cocoa) 模态模式，Cocoa使用这个模式给模态视图投递事件
Event tracking NSEventTrackingRunLoopMode(Cocoa) 这个模式和iOS的流畅不无关系，当我们拖动鼠标或者滑动scrollView的时候，处于这个模式下的run loop会限制其他事件
Common modes NSRunLoopCommonModes(Cocoa) kCFRunLoopCommonModes(Core Foundation) 通用模式，如果run loop运行在这个模式下，等于运行在以上所有模式下

Input Sources

input sources异步投递事件到线程，

当我们在子线程上执行例如-performSelector:withObject:afterDelay:等方法时，该线程必须有一个活跃的run loop，因为这是我们自己创建的，我们必须手动去启动线程

run loop会在每次循环的时候处理全部入队列的selector calls，而不是在每个循环只处理一个

Timer Sources

timer关联到run loop的模式，当timer不再它所安排运行的那个mode时，它并不会被触发直到run loop运行在自己安排的那个线程。相似的，如果timer在一个run loop执行但还没有执行完的时候触发，那么它必须等到下次run loop循环重新触发，如果run loop根本没有运行，那么timer就不会被触发
