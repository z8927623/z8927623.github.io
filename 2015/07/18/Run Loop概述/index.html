Run Loop概述

发表于 2015-07-18   |  

啥是run loop

run loop就是当我们线程有任务的时候唤醒它，然后将任务分配给它去处理；当没有任务的时候，使线程沉睡，节省资源。

每个线程都已经自动关联了run loop，我们不需要去创建它们。但是，并不是每个线程的run loop都是已经运行着的：

  • 主线程：主线程的run loop作为应用启动程序的一部分已经自动运行了
  • 子线程：子线程的run loop是没有运行的，这个时候我们就需要手动开启。

既然run loop有着唤醒线程并让它处理的任务，它就需要监听一些事件，run loop接收两种类型的事件源：

  • 输入源（Input sources）：除timer源外的其他事件源，分为
    • Port-Based Sources：port相关输入源
    • Custom Input Sources：自定义输入源
      • Cocoa Perform Selector Sources：我们常用的perform方法
      • Other Custom Sources：其他自定义输入源
  • Timer源（Timer sources）：熟知的NSTimer，或者CFRunLoopTimerRef

在详细讲解事件源之前，先谈谈run loop mode。

Run Loop Modes

每个run loop可以有一个或者多个mode，但是run loop必须运行在一个mode下。比如主线程的run loop有kCFRunLoopCommonModes， UIInitializationRunLoopMode，kCFRunLoopDefaultMode，GSEventReceiveRunLoopMode，UITrackingRunLoopMode这么多mode，但是它在启动的时候运行在UIInitializationRunLoopMode，在滚动的时候运行在UITrackingRunLoopMode。

一般情况下子线程的run loop运行在默认mode下就OK了，也就是NSDefaultRunLoopMode或kCFRunLoopDefaultMode。我们可以自己设置mode，这些mode包括系统自带的也可以自定义mode。系统mode如下表。

当我们指定mode之后，只有和该mode相关联的事件才能被处理，其他不相关的事件会被挂起，直到run loop回到和该事件对应的mode运行。举个例子，当我们滑动scrollView的时候，run loop是运行在UITrackingRunLoopMode下的，这个模式处理滑动有关的事件而挂起其他事件，如果这个时候恰巧有timer或者网络等事件过来的话，那么它们的回调就不会被处理，直到scrollView停止滚动，这时候run loop重新回到之前mode下，就可以处理这些事件了。

Mode Name Description
Initialization UIInitializationRunLoopMode 启动时候的mode
Default NSDefaultRunLoopMode(Cocoa或Cocoa Touch) kCFRunLoopDefaultMode(Core Foundation) 默认模式，该mode处理输入源（网络事件除外），大部分情况我们使用这个模式就OK了
Connection NSConnectionReplyMode(Cocoa) 这个模式是给NSConnection处理网络回调用的
Event tracking NSEventTrackingRunLoopMode(Cocoa) UITrackingRunLoopMode(Cocoa Touch) 这个模式和iOS的流畅不无关系，当我们滑动scrollView的时候，处于这个模式下的run loop会挂起其他事件而只处理滚动相关事件
Modal NSModalPanelRunLoopMode(Cocoa) 模态模式，Cocoa使用这个模式给模态视图投递事件
Common modes NSRunLoopCommonModes(Cocoa或Cocoa Touch) kCFRunLoopCommonModes(Core Foundation) 通用模式，如果run loop运行在这个模式下，等于运行在以上所有模式下

注意，我们添加的输入源或者timer可以运行在NSRunLoopCommonModes，但是run loop无法运行在NSRunLoopCommonModes，也就是说，[[NSRunLoop currentRunLoop] runMode:NSRunLoopCommonModes beforeDate:[NSDate distantFuture]];这样的代码是没有真正运行了run loop的。因为，run loop一次只能运行在一个确切的mode下，而不是mode集合，但是我们可以标注事件被不同mode的run loop处理。

Timer Sources

timer source我们都知道，有一点需要注意，由于它是基于run loop的，因此它不是实时的，也就是说不是特别准确。

比如我把timer关联到某个mode，这个时候run loop恰巧不在那个mode，那么timer并不会被触发直到run loop回到那个mode。典型的例子就是scrollView上的timer。

相似的，假如这个run loop恰好在执行耗时任务但还没有执行完，这个时候timer事件触发了，那么它只有等到下次被触发才有可能执行，比如我安排timer在现在执行，每隔1秒触发，那么由于run loop执行耗时任务被延迟，可能是过2秒或者3秒之后才被执行。

如果这个run loop压根没有运行，那么timer就不会被触发。例子如上去掉[[NSRunLoop currentRunLoop] runMode:NSDefaultRunLoopMode beforeDate:[NSDate dateWithTimeIntervalSinceNow:10]];之后，timer事件就不会执行。

使用run loop

何时使用run loop

  • 使用ports或者其他输入源和其他线程沟通的时候
  • 在线程上使用timer
  • 使用performSelector...等方法
  • 在线程上执行周期性的任务

如何使用run loop

当且仅当在子线程的时候才有必要开启run loop。并且，一个run loop必须有至少一个输入源或者timer去监听，如果没有的话，run loop会立马退出。

举个例子，假如我们继承NSThread并且在main中执行操作，我们的目的是想让线程不退出，因此开启run loop，但由于我们没有添加任何输入源或者timer，即使开启了run loop也没有用，线程会执行完代码后立刻退出，无法达到目的。

有以下几种选项可以配置run loop

  • 没有限制条件

    最简单，只需要调用[[NSRunLoop currentRunLoop] run];或者CFRunLoopRun();，但是不可靠，因为它会使线程陷入无限循环，只有强制退出run loop。当然我们可以添加或移除输入源或timer，但是无法自定义mode。

  • 设置一个超时时间

    比起啥都不设置，设置一个超时时间是个更好的选择，如-runUntilDate:，-runMode:beforeDate:等。我们设置超时时间之后，run loop会一直运行直到超时或者被强制退出，并且，超时后，我们可以重启run loop。

  • 设定特定mode

    指定mode，给了我们更多的控制权，NSRunLoop的-runMode:beforeDate:或者CFRunLoopRunInMode，我们可以指定mode运行run loop。

如何退出run loop

两种方式

  • 设定一个超时时间

    推荐，指定一个合适的超时时间，当任务完成之后，run loop会自动退出。如-runUntilDate:，-runMode:beforeDate:。

  • 手动停止run loop

    使用CFRunLoopStop手动停止run loop

我们是否可以通过移除输入源或者timer来使run loop退出呢，答案是肯定的，但不是很可靠的一个方法，因为有时候系统也会添加一些我们不知道的事件源去处理一些事情，那么我们可能无法全部移除事件源，也就退出失败了。
