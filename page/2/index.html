我们开发iOS或者OSX程序时候，调试是避免不了的，而调试最大的一个利器就是断点。接下来我介绍一下Xcode一些常用的和不常用的技巧。

高级log术

经常在调试程序的时候，我们需要在程序里面打印一些东西，最常用的就是NSLog，但是NSLog有几个缺点

  • NSLog会拖慢运行速度，效率低
  • 大量log导致代码混乱
  • 最重要的一点，假如我们在程序运行的时候想要打印某些信息，这个时候我们可能会想到使用po，这个在打印单个信息的时候很有用，但是比如在上面我们的循环里面，我需要知道每次偶数的时候的idx是多少，那就不得不在编写完后重新编译运行，很麻烦。

那么怎么办呢，还是在条件断点里面设置

我们首先设置条件为偶数，然后再在Action里面选择Debug Command，这和允许我们直接使用lldb或者gdb的命令进行调试，这里我们使用po idx打印出idx的值，表示当满足偶数条件的时候打印。接下来最重要的就是勾选下面那个Automatically continue after evaluating选项，表示在执行动作之后不暂停程序继续运行。

可以看到，Xcode乖乖地打印除了0到10000的偶数。

假如我们需要输出一些信息，也好办，点击Action右边的加号，添加动作，这次我们选择Log Message，这里可以写下我们需要输出的信息，可以选择在控制台输出或者语音播放，或者也可以再添加一个动作，选择Sound，内置丰富的声音，在您调试的时候也能够享受美妙的交响乐。

查看输出，同时伴有frog的叫声

ARC与MRC必知必会

发表于 2016-01-26   |  

我们习惯将ARC的每个字母分开读即读A R C，但是实际上正式的叫法叫ɑ:k，即啊克。好了，下面开始进入正题，谈谈ARC与MRC有什么不同。

ARC下的autorelease pool

ARC下最明显的就是对autorelease pool的改进。

Autorelease pool会自动管理对象的生命周期，每个发送autorelease消息的对象不会马上被释放，它们会被加入到NSAutoreleasePool中，当pool被drain的时候，它会依次给池子中的每个对象发送release消息。因此，使用autorelease管理内存的对象的生命周期被延长了。

MRC下main函数是这么写的

ARC下，我们不需要直接管理autorelease pool了，使用@autorelease即可，并且比NSAutoreleasePool更高效。

那么@autoreleasepool到底干啥了呢，文档上是这样说的

On entry, an autorelease pool is pushed. On normal exit (break, return, goto, fall-through, and so on) the autorelease pool is popped

意思是说在@autoreleasepool入口处，一个autorelease pool会被push进去，出口处会被pop出来。实际上通过断点查看反汇编代码可以发现，所谓的push是调用了objc_autoreleasePoolPush，pop是调用了objc_autoreleasePoolPop。

ARC下的属性修饰符

ARC新增了几个属性修饰符

  • __strong

    ARC下，默认所有的栈变量都是strong类型，这个属性和retain一样，并且假如我们没有为其赋值，会自动初始化为nil。

    NSString *name;
    实际上它等于拥有了以下属性
    __strong NSString *name = nil;

  • __weak

    不会增加对象的引用计数，当指向的对象被释放时，自动置为nil。ARC下替代MRC下delegate的assign方法，既可以打破循环引用又可以避免野指针造成的crash。

  • __unsafe_unretained

    字面意思unretained表示不会增加对象的引用计数，unsafe表示当指向的对象被释放时，该指针就成为野指针了，因此是不安全的。

ARC下的block

ARC和MRC block的类型还是有所区别的。

ARC下创建的block，和其他对象一样，默认是strong类型。当__NSStackBlock__类型的block使用外部变量时，这个block会被copy进堆中，就成为了__NSMallocBlock__（有个例外，直接在参数中传递的block）；如果没有使用外部变量，则是__NSGlobalBlock__。那么有没有办法使它成为__NSStackBlock__呢，可以显式声明block为weak。

ARC

例外，ARC下，如果使用了外部变量的block直接作为函数参数，这个时候还是__NSStackBlock__

讨论完ARC和MRC下block的类型，接下来我们来看看ARC中有哪些需要注意的地方以及和MRC相比ARC为我们所做的工作。

先看函数返回值为block的情况。

MRC下，copy是因为这里创建的block是个__NSStackBlock__（因为它使用了外部变量i），不copy的话当函数返回时它就被释放了，因此必须将它copy到堆上，也就是成为__NSMallocBlock__。实际上如果我们创建的是NSGlobalBlock，这里是可以不用copy的，但是copy总是正确的；autorelease的原因和前面一样，返回对象之后我们必须放弃对其所有权，这样才会被安全释放。

ARC下会自动将返回的栈block copy和autorelease，我们不需要手动书写。实际上这个block在ARC下是__NSMallocBlock__，实验结果来看ARC并没有做copy。

ARC下的循环引用问题

使用block的时候，当A持有block，而block又持有A的时候，就会造成循环引用而相互无法释放。
MRC下我们只需要使用__block修饰对象，block就不会retain它

但是在ARC下，即使你的变量使用了__block，仍旧会被retain，因此仍然会循环引用，这个时候就应该使用__weak或者__unsafe_unretained（iOS5以下）了，但是__unsafe_unretained是不安全的，因为当它指向的对象被释放的时候，而这个指针还指着那块被释放的地址，也就是野指针，就会造成crash；而__weak则会在发生这个情况的时候自动将指针置为nil，OC里对nil发消息是可以的。

以下是几点需要注意的循环引用的地方

  • block中使用了属性定义的变量，如self.xxx，block就会retain self，而如果self又retain该block，就会造成循环引用，也是最常见的类型。

  • block中使用了不是使用属性定义的成员变量，默认是strong类型，也会被强引用。这个时候我们就可以使用一个临时变量来存储。

  • delegate的属性修饰符要从assign改为weak，这样既可以打破循环引用又可以避免野指针造成的crash。

当然也并不是一有block我们就要使用弱引用，以下是几个不需要使用weak的地方

  • 当block不被诸如vc之类的持有时，我们不需要使用weak，比如我定义了一个函数使用block作参数，但是这个block仅仅是参数而没有被我持有。

  • GCD中一般情况下不需要使用weakSelf，GCD中block虽然retain了外部的变量，但是我们并不持有该block

  • 系统的一些诸如动画之类的block，原因同上

关于ARC下__block和__weak的区别

ARC下的一些开源库中我们经常能看到__block和__weak一起修饰某个变量，他们两个属性各有各的作用，__weak是为了避免循环引用；而__block是为了可以修改外部的变量，不是避免循环引用，因为ARC下__block仍然可以retain变量，不要搞混淆了。
