iOS Method Swizzling

发表于 2015-09-20   |  

先来看下面这段代码

这段代码想必只要我们学习过method swizzle的同学都很熟悉，实现也比较简单。其中有两个地方需要注意，也是难点，即

BOOL didAddMethod = class_addMethod(class, oldSelector, newIMP, newMethodType);

和

[self xxx_viewDidLoad];

讲解第一点之前，必须先了解一下class_getInstanceMethod函数的功能。

其中有个需要注意的地方，官方文档描述该方法如下

Discussion

Note that this function searches superclasses for implementations, whereas class_copyMethodList does not.

从中可以看到，该函数会搜索父类的实现，如果该选择器没有在本类实现，则它会查找父类的实现，如果父类实现了，则返回；而class_copyMethodList并不会返回父类的方法。

因此我们回到刚才那个问题，假如我们要替换的方法在本类中并没有实现而父类实现了，则class_getInstanceMethod返回了父类的实现，我们再使用method_exchangeImplementations交换实现，这样的结果是什么呢？就是我们替换了我们想要替换的类的父类的方法，这并不是我们想要的。

因此先用class_addMethod这个方法Hit-Testing一下本类有没有实现这个方法，

  • 如果本类已经实现了该方法，则会添加失败，那么我们开开心心的method_exchangeImplementations就好了；
  • 但是假如本类没有实现呢，那么该方法会自动为该类添加该方法，并且指向我们的新实现，然后我们再使用class_replaceMethod将新方法指向老的实现。实际上因为class_addMethod已经成功，即已经oldSelector --> newIMP，所以我们只需要newSelector --> oldIMP即可，因此class_replaceMethod可以被替换成method_setImplementation(swizzledSelector, method_getImplementation(originalMethod));，直接将新方法指向老实现。

可能有些同学会被绕晕，我举个实例。

比如我想要替换掉UITableViewController的viewDidLoad的实现，由于UITableViewController的viewDidLoad是在其父类UIViewController中实现的（class-dump出来看到的），因此class_addMethod会添加成功；如果直接交换实现的话，就把它的父类UIViewController的viewDidLoad的实现替换了，这就不是我们的初衷了。

接下来讲一下为什么要调用老方法，或者我们不调用老方法那它真的就没调用了吗。

顾名思义，系统调用老方法的时候，由于它的实现被指向了我自己的实现，我们实现自己的逻辑之后，当然要调用原来的方法，不然就像隐藏了一个定时炸弹，指定什么时候出现什么奇怪的问题。比如我自己为了测试，hook了UIView的-setBackgroundColor:方法，假如我不调用老方法，则不管我将view设置成什么颜色，它将都是黑色，这说明了的确老方法的实现被我们绕过了。因此，为了安全起见，我们在自己的方法里面一定要调用原来的方法。
